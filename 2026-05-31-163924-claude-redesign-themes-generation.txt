  Rust Async Patterns

  ūüü° find_available_port blocks the async runtime
  bonsai.rs:99‚Äď107 ‚ÄĒ TcpListener::bind is called in a synchronous loop directly
  inside the async command handler bonsai_start_server. Even though it's fast,
  blocking syscalls must be wrapped in spawn_blocking per Tokio's rules.
  Everything else in the file follows this correctly (e.g., kill_port_sync).

  // Current ‚ÄĒ blocks runtime thread
  let port = find_available_port(config.port, MAX_PORT_OFFSET)...

  // Fix
  let base_port = config.port;
  let port = tokio::task::spawn_blocking(move || find_available_port(base_port,
  MAX_PORT_OFFSET))
      .await
      .map_err(|e| bonsai_error(...))?
      .ok_or_else(...)?;

  ūüü° TOCTOU race in bonsai_schedule_stop
  bonsai.rs:491‚Äď511 ‚ÄĒ Two concurrent bonsai_schedule_stop calls can both read
  has_timer = false, both spawn a timer task, and both store theirs ‚ÄĒ the first
  spawned task gets orphaned and leaks, causing double bonsai:stop-timeout
  emissions. The window between the two lock acquisitions is the problem.

  Fix: merge the check-and-set into a single lock scope:
  let mut bonsai_guard = state.bonsai_process.lock().await;
  if let Some(ref server) = *bonsai_guard {
      if server.stop_timer.is_some() { return Ok(()); }
  }
  let stop_handle = tokio::spawn(async move { ... });
  if let Some(ref mut server) = *bonsai_guard {
      server.stop_timer = Some(stop_handle);
  }

  ūüĒĶ Weak PNG size validation
  bonsai_assets.rs:163 ‚ÄĒ png_bytes.len() < 1024 as a validity check is fragile.
  A 1√ó1 PNG is ~67 bytes; a real 512√ó512 will be much larger. But it also means
  a 1025-byte JSON error response from the server passes this check. Better to
  sniff the PNG magic bytes: png_bytes[..8] == [137, 80, 78, 71, 13, 10, 26, 
  10].

  ‚úÖ Lock-across-await hygiene is excellent
  The careful lock ordering comments and the "take out, drop, then await"
  pattern throughout bonsai.rs and bonsai_assets.rs are correct and
  well-documented. The bonsai_generation_token using std::sync::Mutex (not
  tokio) and never holding it across .await is right.

  Python / Bonsai alternative

  No viable Rust path exists for this workload. The Bonsai Image Demo runs FLUX
  with GemLite quantization via Apple's MLX framework
  (mlx-community/bonsai-image-4B-ternary-gemlite). MLX is Python-only (no Rust
  bindings). The candle crate supports Metal but has no GemLite quantization
  support. The current architecture ‚ÄĒ Rust managing a Python uvicorn subprocess
  and calling it over HTTP ‚ÄĒ is already the right approach and produces clean
  separation. Nothing to change here.

  ūüĒī CRITICAL: Hotspot creation fails silently for old projects ‚ÄĒ 
  createHotspotWithLink throws "Screen not found"

  navigation.ts:237:
  const screen = nav.screens.find((s) => s.id === screenId);
  if (!screen) throw new Error(`Screen ${screenId} not found`);

  addScreenToNavigation is only called from SidebarRail.tsx when a screen is
  created via the sidebar. If the user opened an older project that predates
  navigation.json, or if navigation.json got corrupted/deleted, screens exist on
  disk but are absent from nav.screens. Clicking "Select Element" to create a
  hotspot then throws and the error toast shows "Failed to create hotspot:
  Screen X not found". The fix is to call addScreenToNavigation as a fallback
  inside createHotspotWithLink when the screen isn't found:

  // instead of throwing:
  let screen = nav.screens.find((s) => s.id === screenId);
  if (!screen) {
    nav.screens.push({ id: screenId, path: `/${screenId}`, title: ..., ports:
  getDefaultPorts(screenId) });
    screen = nav.screens[nav.screens.length - 1];
  }

  ūüĒī CRITICAL: Duplicate links can be created ‚ÄĒ link ID format inconsistency

  createHotspotWithLink (navigation.ts:268) creates links with id: 
  \link-${Date.now()}`. addNavLink (navigation.ts:123) creates links with id:
  `${from}:${fromPort}->${to}:${toPort}`and deduplicates usingnav.links.some((l)
  => l.id === id)`. Since the two formats never match, the same connection
  drawn in FlowsView after a hotspot was created produces a duplicate link in
  navigation.json. The router then emits duplicate navigator entries pointing
  the same selector at the same path ‚ÄĒ effectively a no-op but still corrupts
  the data model.

  Fix: use the same ID format in both places. Change createHotspotWithLink to
  use the canonical format:
  const linkId = `${screenId}:${portId}->${targetScreenId}:${targetPortId}`;
  if (!nav.links.find((l) => l.id === linkId)) {
    nav.links.push({ id: linkId, ... });
  }

  ūüü° Editing a hotspot highlight IS editing a route ‚ÄĒ by design, but not 
  communicated

  The user's observation is correct. In the current model, hotspots and ports
  are the same object: a hotspot is just a { portId, selector, rect } overlay on
  a port in nav.screens[i].ports. Editing a port's name in PortsEditor renames
  the route handle in FlowsView. Deleting a port in PortsEditor removes both the
  FlowsView handle AND the corresponding hotspot highlight overlay.

  This is architecturally sound but unintuitive. The PortsEditor header says
  "Screen Ports" ‚ÄĒ it should show when a port has an associated hotspot and make
  clear that renaming/deleting it affects the highlight too. At minimum, show a
  MousePointerClick icon next to ports that have hotspot entries.

  ūüü° PortsEditor's built-in "Pick element" button is never shown in ScreensPanel

  ScreensPanel.tsx:888‚Äď893 passes neither onSelectElement nor isSelectingElement
  to PortsEditor. The element-selection button inside PortsEditor is dead code
  in this context. The toolbar button above the code pane (lines 857‚Äď876)
  handles the flow instead, which works but means PortsEditor has an unreachable
  code path.

  ūüü° FlowsView node deletion is half-wired

  FlowsView.tsx:378 sets deleteKeyCode="Delete", which removes nodes from local
  React Flow state. But there's no onNodesDelete handler calling
  removeScreenFromNavigation. Pressing Delete on a screen node in FlowsView
  silently orphans its navigation data ‚ÄĒ the node comes back on reload. Add an
  onNodesDelete handler or disable node deletion entirely (deleteKeyCode={null})
  since screen lifecycle is managed through the sidebar.

  ūüĒĶ FlowsPanel never dispatches navigation-changed after node position drags

  FlowsView.tsx:291‚Äď303 ‚ÄĒ onNodeDragStop saves position to navigation.json but
  does NOT dispatch window.dispatchEvent(new Event("navigation-changed")). If
  ScreensPanel is listening for position changes to update its port list (it
  isn't, but it does listen for nav changes in general), this can cause stale
  state. Minor but inconsistent with the rest of the codebase.

# Tier 2 — Source-Location Mapping for Annotation Targeting

## 1. Goal

Tier 1 (implemented) resolves an annotation to a **structural CSS selector**
(`main > section:nth-of-type(2) > button`) plus tag/text, via the existing
`find-element-at` postMessage bridge in
`src/lib/scaffold-shadcn/main-template.ts`. The model has to *find* the
matching JSX by reading the screen file and pattern-matching tag/structure/text.

Tier 2 would resolve an annotation directly to a **source location** —
`file:line:column` (or a stable element ID that maps to one) — so the model
can jump straight to the right JSX without searching. This doc covers how
that would actually be built in Prototyper specifically.

## 2. Prototyper's scaffold/build pipeline (what we control)

- `vite.config.ts` for every generated app is fully generated by
  `getGeneratedViteConfig()` in `src/lib/scaffold-shadcn/templates.ts:104-144`.
  It's a plain string template — `plugins: [tailwindcss(), react()]` via
  `@vitejs/plugin-react`, plus a path alias and optional API proxy. We own
  this file completely and can add a Babel plugin to the `react()` call with
  a one-line template edit.
- `constants.ts` already has the precedent for *programmatic vite.config.ts
  patches*: `patchViteFsAllow()` and `patchViteResolveDedupe()` both do
  regex-based merges into an existing `vite.config.ts` (idempotent, safe to
  re-run). A Tier 2 patch (`patchViteBabelSourceStamp()`?) would follow the
  same pattern.
- No React/Vite/Babel versions are pinned by Prototyper — they come from
  whatever `shadcn@4.10.0 init -t vite` resolves at scaffold time (currently
  React 19, Vite 6+, `@vitejs/plugin-react` latest).
- Dev server: `bun_dev()` (`src-tauri/src/commands/process.rs:175-178`) just
  runs `bun dev --port <port>` — i.e. the generated project's own Vite dev
  server, unmodified by Prototyper at runtime. Any instrumentation has to live
  in the generated `vite.config.ts`/Babel config, not in how we launch it.
- `export_project`/`export_component` (`src-tauri/src/commands/export.rs`)
  copy source files **as-is** into a zip — no build step, no transform pass.
  A dev-pipeline Babel plugin therefore **never touches on-disk `.tsx`
  files** and needs **no explicit stripping** for exports — it only affects
  what Vite serves/builds, which exports don't consume.

**Net effect**: Tier 2 is purely a `vite.config.ts` / Babel-plugin-config
change at scaffold time (and a one-time patch for already-scaffolded
projects, same as any other `constants.ts` patch). No Rust/export changes
needed.

## 3. Why `__source` / `_debugSource`-based tools don't fit

`@vitejs/plugin-react` supports `babel: { plugins: [...] }` for custom Babel
plugins — that part is easy. The hard part is *which* plugin and *what it
stamps*.

- **`babel-plugin-transform-react-jsx-source`** (the classic "where did this
  element come from" plugin) injects a `__source` **prop** on each JSX
  element. React used to copy this into `fiber._debugSource`, which devtools
  and tools like `react-dev-inspector` read at runtime.
- **React 19 removed `_debugSource` from the Fiber** (facebook/react#29092,
  #32574). `__source` props are no longer mirrored onto the fiber by default,
  so anything that reads `fiber._debugSource` (the whole
  react-dev-inspector/Locator.js family) is broken or requires non-default
  patches.
- Conclusion: **don't use fiber/devtools introspection.** Any Tier 2 plugin
  must emit a **plain DOM `data-*` attribute** directly in the JSX output
  (e.g. `<button data-oid="a3f9">`), which `document.elementFromPoint()` can
  read with zero React-internals dependency — exactly how `getSelector()`
  already works for Tier 1, just with a richer payload.

## 4. The Onlook pattern (verified from source) — recommended design

Fetched Onlook's `packages/parser/src` (onlook-dev/onlook, MIT). Their
approach cleanly solves the "line numbers go stale after every edit" problem
that Prototyper would otherwise hit immediately (Prototyper's `edit_file` —
`src-tauri/src/agent/executor.rs:388-527` — is a string-based find/replace on
the existing file, not an AST rewrite, so any cached `line:column` becomes
wrong the moment the model edits the file).

**Two-part design:**

1. **Stable opaque ID, stamped at build time** — a Babel/AST transform adds
   `data-oid="<random-id>"` to every JSX element (Onlook's `createOid()` +
   `createOidAttribute()`). This ID is *content-independent* — it doesn't
   encode a line number, so editing the file doesn't invalidate it... **but**
   it does need to be (re-)stamped into the source file itself for the ID to
   survive edits (see open risk below — this is where Onlook's model diverges
   from Prototyper's).
2. **On-demand AST → position map, built fresh each time** —
   `createTemplateNodeMap({ast, filename})` re-parses the *current* file and
   walks the AST, building `Map<oid, {path, startTag: {line, column}, endTag,
   component}>`. Because this map is rebuilt from the live file on every
   lookup, it's never stale — only the `oid → position` resolution is
   recomputed, not the `oid` itself.

**Adapted for Prototyper:**

- `find-element-at` (in `main-template.ts`) already runs inside the iframe
  and has `el` — extend the `hotspot-created` reply with
  `dataOid: el.closest('[data-oid]')?.getAttribute('data-oid')` alongside the
  existing `selector`/`elementTag`/`elementText`.
- **A new Rust/agent-side step** is required that Tier 1 didn't need: given
  `dataOid` + the screen's file path, locate the JSX element with that
  `data-oid` in the current source and report its `line:column` to the model.
  This is the `createTemplateNodeMap` equivalent — an AST walk over the
  screen's `.tsx` file, run **on the host side** (not in the iframe), e.g. as
  a new lightweight tool (`find_jsx_by_oid` or similar) the model can call, or
  pre-resolved by the frontend before sending the annotation context.
  - Prototyper would need a JS/TS AST parser available to either the Rust
    backend (via a bundled `swc`/`oxc` crate — both have Rust-native AST
    parsing, no Node dependency) or a small Node/bun-invoked helper script
    using `@babel/parser` + `@babel/traverse` (consistent with the
    already-Babel-based stamping plugin).

## 5. Recommended approach: wrap `react/jsx-dev-runtime` via Vite alias (zero new deps)

Rather than adding a Babel plugin (a new devDependency + a new transform
stage), reuse a transform **`@vitejs/plugin-react` already performs**.

### How it works

In dev mode (`vite dev`), `@vitejs/plugin-react` compiles every JSX tag via
`@babel/plugin-transform-react-jsx-development`, producing calls like:

```js
_jsxDEV("button", { children: "Get Started" }, void 0, false,
  { fileName: "/abs/path/src/pages/home.tsx", lineNumber: 42, columnNumber: 8 },
  this)
```

That 5th argument (`source: {fileName, lineNumber, columnNumber}`) is emitted
by Babel **for every JSX element, in every React version, including React
19** — confirmed via [babel/babel#17571](https://github.com/babel/babel/issues/17571).
In React 19.2 specifically, `react/jsx-dev-runtime`'s own `jsxDEV` export
dropped the `source`/`self` parameters from its signature (verified directly
in the story-book project's
`generated/node_modules/react/cjs/react-jsx-dev-runtime.development.js:324` —
`exports.jsxDEV = function (type, config, maybeKey, isStaticChildren) {...}`,
4 params only) — but Babel **still passes** the extra arguments; React's
`jsxDEV` simply ignores them (harmless extra JS args).

So the exact source location is already flowing through every dev build —
it's just discarded before reaching the DOM. The fix: **intercept it before
React does.**

1. Add one generated file, `src/dev/jsx-dev-runtime-shim.ts`, that re-exports
   `Fragment` and wraps `jsxDEV`: for string-type elements (real DOM tags,
   not components) it reads the `source` arg and injects
   `data-source-loc="src/pages/home.tsx:42:8"` (path relative to `src/`) into
   `props`, then delegates to the *real* `jsxDEV`.
2. In `vite.config.ts`, alias `react/jsx-dev-runtime` → this shim, and add a
   second alias `react/jsx-dev-runtime-real` → the actual resolved path to
   React's real `jsx-dev-runtime.js` (via `createRequire(import.meta.url).resolve(...)`),
   so the shim can import the real implementation without aliasing itself
   into infinite recursion. Both aliases are gated to `command === "serve"`
   (dev only) via the function form of `defineConfig`.
3. `find-element-at` in `main-template.ts` then reads
   `el.closest('[data-source-loc]')?.getAttribute('data-source-loc')` — same
   shape as the Tier 1 selector, no AST, no Rust changes, **no new
   dependencies** (only `react`, `@vitejs/plugin-react`, `vite`, already
   present).

### Trade-offs (same as the Babel-plugin variant, §5 prior draft)

- ✅ Zero new deps — one generated shim file + a `resolve.alias` addition,
  following the exact pattern of `patchViteFsAllow`/`patchViteResolveDedupe`.
- ✅ `lineNumber`/`columnNumber` point at the element's **opening tag**, exact
  to the column, as of the file's last Vite transform (last save/HMR).
- ❌ Staleness window between annotation-creation and send-time if the model
  edits concurrently — mitigate by re-running `find-element-at` at send-time
  (in `serializeAnnotations`'s caller), not caching the click-time value.
- ❌ `.map()` lists: every rendered item shares one JSX source location — same
  limitation as Tier 1's `nth-of-type` selector, not worse.
- ❌ Shared components: clicking inside `<Card>` resolves to the location
  *inside `card.tsx`*, not the `<Card>` usage site — same caveat as Tier 1.
- ⚠️ **New risk specific to this approach**: aliasing `react/jsx-dev-runtime`
  could interact with `@vitejs/plugin-react`'s Fast Refresh machinery, which
  may do identity/module-boundary checks against that specifier. Needs to be
  verified empirically (does HMR still work correctly with the alias in
  place?) — this is the main thing the story-book spike should confirm.

**A live spike of this approach has been created in the `story-book` test
project** (`~/.local/share/com.m.prototyper/projects/story-book/generated/`):
`src/dev/jsx-dev-runtime-shim.ts` + a `vite.config.ts` alias. If it proves
unreliable (e.g. breaks Fast Refresh), fall back to the Babel-plugin
`data-loc` variant described in the original draft of this section — same
DOM-attribute shape, same downstream touchpoints, just sourced via a Babel
plugin + new devDependency instead of a runtime wrapper.

## 7. Open risks / questions

- **Fast Refresh interaction unverified**: aliasing `react/jsx-dev-runtime`
  to a wrapper module is, as far as this research found, not a documented
  pattern — `@vitejs/plugin-react`'s Fast Refresh transform may special-case
  that module specifier for HMR boundary detection. The story-book spike
  (`projects/story-book/generated/`) needs to confirm both that
  `data-source-loc` appears in the DOM **and** that HMR/Fast Refresh still
  works correctly with the alias in place. If Fast Refresh breaks, fall back
  to the Babel-plugin `data-loc` variant (a custom ~30-line
  `JSXOpeningElement` visitor is preferable to a third-party package, per the
  earlier finding that community plugins for this are sparse/unmaintained).
- **`.map()` / list-item ambiguity**: as noted in §5, a single JSX source
  location is shared by every rendered item of a `.map()`. Tier 2 doesn't
  resolve this any better than Tier 1's `nth-of-type` selector. If this
  becomes a real pain point, Onlook's oid approach (§4) is the actual fix —
  but it requires per-render unique IDs, which a static `data-loc` (same
  string for every iteration) cannot provide.
- **Re-stamping on edit**: if Tier 2 ever needs *stable* IDs (oid-style, §4)
  rather than line-based locs, those IDs must be written into the `.tsx`
  source itself (so they survive Vite's re-transform), which means
  `edit_file`/`write_file` calls that touch JSX would need to either preserve
  existing `data-oid` attributes verbatim or have them re-stamped by a
  post-write pass — meaningfully more invasive than the `data-loc` variant,
  which needs no source-file changes at all (the attribute is injected purely
  by the Babel transform, never written to disk).
- **Dev vs. build parity**: confirm the chosen Babel plugin runs in both
  `vite dev` (what the iframe preview uses) and `vite build` (what
  `bun_build`/production bundles use) — or explicitly gate it to dev-only via
  `apply: 'serve'` in the plugin config if it should never affect production
  builds of the generated app (it has no reason to ship to end-users of the
  generated app).
- **Multi-file screens / shared components**: if a screen imports shared
  components (e.g. `src/components/ui/card.tsx`), a click inside a `<Card>`
  resolves to a `data-loc` pointing at `card.tsx`, not the screen file the
  model is iterating on. The model needs to understand `loc` may point at a
  shared component file, not always the active screen — same caveat applies
  to Tier 1 selectors but is more pronounced here since `loc` looks more
  "authoritative".

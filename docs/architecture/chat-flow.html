Multi-tool iteration (read_file → write_file)

If the model calls read_file before write_file, the agent loop does not break after the tool result — wrote_file remains false. It rebuilds ChatMessageRequest::new(model, vec![]) (empty messages, history is already populated) and calls stream_turn() again for iteration 1. The history at that point contains: [system, user…, assistant{tool_calls:[read_file]}, tool{contents}] — with tool_calls correctly preserved in the assistant message (unlike send_chat_messages_with_history_stream which drops them). The model receives full, valid context for the follow-up turn. MAX_ITERATIONS = 10.

Source: agent_loop.rs:140–212 — the if wrote_file { break } guard, continuation request with vec![] messages, MAX_ITERATIONS check
Architecture notes

Model capabilities gating: useChat.ts uses useModelCapabilities to gate thinkEnabled and toolsEnabled states. If the model doesn't support tools, effectiveOutputPath is undefined and the direct HTTP path is used instead of the agent loop.

Write-file path hard-coding: execute_write_file intentionally ignores the model-provided filename and always uses the caller's output_path. This prevents the model from writing files outside the project directory (path-traversal guard via .. check). Parent directories are created automatically.

Cancellation: stop_generation_stream signals a CancellationToken which causes the stream loop to break via tokio::select!. For Ollama, dropping the HTTP connection is the standard cancellation pattern — there is no /api/abort endpoint.

History fix: The agent loop manually constructs assistant messages with tool_calls attached before pushing to history. The ollama-rs crate's built-in history helper (send_chat_messages_with_history_stream) drops tool_calls, which breaks multi-turn tool conversations. The fix is applied at agent_loop.rs:78–80.
